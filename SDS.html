1.Introduction

The aim of the project is to build a document management software which digitizes existing documents/files from the Department of Finance. The converted digital format will be stored into storage defined by the Department of finance using a simple document system capable search and retrieval for attaching the same.It will be accessible via web browser as well as phone application.
The system will be user friendly and easily accessible, store, manage, edit, send and delete documents.The system will allow users to create accounts, upload unlimited files and search for required files.It is based on web-application and Relational Database. Database server will be there for handling huge data form different city branches of the organization. As everything is digitized so it will be less time consuming and efficient than typical existing manual system. We vision to provide best possible software system with low pricing and easy-to-use application.

3. Description of Design Components

3.1 Server-side

The database in our server-side is made using firebase, and the backend server is Django. The server is a MVC type architecture where the Controller handles any incoming request from the REST API Framework, receives information from the models and return response to be displayed through a view.

In the server-side, the IT department will have the following responsibilities:

  1. Backing up the files: If a user uploads a file, it will be automatically stored inside a folder in the server-side. There will be two folders for this, one labelled public and the other private. The IT department will have to back up these folders frequently incase the system crashes or encounters any problem.
  2. Maintenance: The IT department will be responsible for maintaining the website and the android application from time to time, and look for bugs in the system.
  3. Security: The IT department will be responsible for enforcing security to avoid data breach.


3.2 Client-side

The client side of our design contains HTTPS requests from Android and Web. All requests are passed through a REST API Framework that fetches the required information and displays it through appropriate format.

The client be able to visit the application using different devices (Android and Web). The views will be different according to the device, but the functionalities remain the same.

  1. Login and Registration page: The user will be required to register, or login with his credentials if he is already registered. In order to gain access to the application this is necessary.

    Android view:

  2. Homepage: After the user logs in, he will be directed to the homepage. The homepage has features depending on the type of user (Ex. The administrator has a button which redirects him to the control page, where he can restrict user, or give him access to a private file).
  3. Scanner: We will not be making a scanner from scratch, but will use a library to provide this feature. The users can utilize the scanner to take pictures of documents, which will then be automatically converted into digitalized versions. The user can then choose to save the file.
  4. Control Page: This page is only accessible via certain types of users (administrator and manager). Through this page the officials can look into user activities, ban user accounts and give permission to access private files.

    • 8. Implementation Plan

    We plan to execute the project in the following steps:

    1. Overall Design of the System: This is the planning period of our project. We have created class diagram, sequence diagram, ER diagram and tried to break down the project for ease of execution. We will later utilize these diagrams to visualize our project. The final product might not be as predicted, but this will surely guide us to reach the oucome we desire.
    2. Frontend: First we are going to create the frontend. We are going to create the frontend of the website using html, css and javascript, whereas we will be using android studio for the frontend of the android.
    3. Backend: After the front end is created, we are going to create the database using firebase. The database might not be identical to what we are proposing through the ER diagram, but it will be very similar. Then we will have to connect the website frontend with Django backend server.
    4. API Framework: We will be requiring to create an API Framework to connect the database with the frontend of the android application
    5. Improvise: We will have to test the applications multiple times under various conditions to find any bug in the system. We will also aim to improve it and keep it updated.
